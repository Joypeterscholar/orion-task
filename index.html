01 Task One
How GitHub Actions
actually work
📁
01

The Workflow File

Everything starts with a YAML file placed in .github/workflows/. This file defines triggers, jobs, and steps. GitHub reads it automatically on every matching event.

⚡
02

Triggers (on:)

Workflows fire on events — push, pull_request, schedule, or manual dispatch. You control exactly what kicks off the pipeline.

🏃
03

Runners

Jobs run on virtual machines called Runners. GitHub provides Ubuntu, macOS, and Windows runners. They're fresh, isolated environments spun up per job.

🔗
04

Jobs & Steps

A workflow contains jobs (parallel by default). Each job has sequential steps — shell commands or pre-built Actions from the marketplace.

📦
05

Actions & Marketplace

Reusable units of work. Use uses: actions/checkout@v4 to check out code, or pick from thousands of community Actions on the GitHub Marketplace.

🚀
06

Secrets & Deploy

Store tokens and keys as encrypted Secrets in your repo settings. Reference them as ${{ secrets.MY_KEY }} to deploy securely without exposing credentials.

📤

Developer pushes code

A git push to the main branch fires the push event trigger on GitHub.

🔍

GitHub reads workflow YAML

The platform parses .github/workflows/ci.yml and determines which jobs to schedule.

🏗️

Runner is provisioned

A fresh virtual machine spins up with the specified OS. The repo is checked out automatically.

🧪

Steps execute sequentially

Install dependencies → run linter → run tests → build artifact. Each step reports pass or fail.

✅

Deploy on success

If all steps pass, the final deploy step pushes to production. Failure stops the pipeline and notifies the team.

02 Task Two
Railway vs
Amazon Web Services
Railway Developer-first PaaS
🎯

Target User

Indie devs, startups, side-projects. Built so developers can ship without ops knowledge.

⚙️

Setup Complexity

Connect GitHub repo → done. Zero config deployments. Auto-detects your stack.

💰

Pricing Model

Usage-based. Free tier available. Scales with what you consume — simple and transparent.

🌍

Infrastructure Control

Managed abstraction. You don't pick regions, VPCs, or load balancers — Railway handles it.

📊

Scalability

Great for small-to-medium scale. Not designed for massive enterprise workloads.

🔌

GitHub Actions Integration

Native — deploy from Actions using the Railway CLI or official Action with one step.

AWS Enterprise Cloud Platform
🏢

Target User

Enterprise teams, large-scale SaaS, orgs needing compliance, fine control, and global reach.

🧩

Setup Complexity

Steep learning curve. IAM, VPCs, Security Groups, Load Balancers — extensive configuration required.

💳

Pricing Model

Pay-as-you-go with hundreds of services. Costs can surprise — requires billing monitoring and budgets.

🌐

Infrastructure Control

Full control. 200+ services. Pick regions, AZs, instance types, networking, storage — everything.

📈

Scalability

Virtually unlimited. Powers Netflix, Airbnb, NASA. Handles any scale with proper architecture.

🔌

GitHub Actions Integration

Via OIDC or access keys. Deploy to EC2, ECS, Lambda, S3 — official AWS actions available.

⚖️

When to choose which

Railway — perfect for getting a project live fast with zero ops overhead. Ideal for MVPs, demos, and developer tools where shipping speed beats infrastructure control.
AWS — the right choice when you need global scale, compliance requirements, fine-grained security, or deep integrations across a complex stack. Worth the complexity at scale.
